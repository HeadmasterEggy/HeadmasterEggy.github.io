INFO5990 Week 5 Tutorial Summary

Objectives

本次 tutorial 的目标是帮助学生将 communication（沟通）、teamwork（团队合作）和 stakeholder management（利益相关者管理）的概念应用于实际的 IT 项目场景。通过 stakeholder mapping、案例讨论等互动活动，学生将增强有效沟通、多元团队协作和利益相关者管理的能力。完成后，学生应能识别 IT 项目中的常见挑战，并练习应对策略，理解这三项技能对专业成功的重要性。

Learning Outcomes

完成本次 tutorial 后，学生应能够： - LO3 — 分析与 IT 专业实践相关的更广泛问题，并了解 IT 行业人力资源管理的当前趋势 - LO5 — 展示构建有效书面和口头沟通的能力，包括冲突解决、合同谈判、团队组建、领导力和团队动态 - LO6 — 应用研究方法、模型和工具分析 IT 专业实践，识别和批判变化中的信息，并判断结果的潜在有效性

Part A — 7Cs of Communication

原句

“Hey, project’s late, not sure when it’ll be done. Will let you know.”

改写后

The project is currently behind schedule because some development issues are still being resolved. At this stage, we expect it to be completed by next Wednesday, although this may change depending on testing results. I will provide a confirmed update by Friday.

说明

这句话原本的问题是太随意，也不够清楚。改写以后，信息更完整，语气也更专业。新的表达说明了项目为什么延期、预计什么时候完成，以及下一次更新时间是什么时候。这样更符合 7Cs of Communication 的要求，也就是 clear、concise、complete 和 courteous。

Part B — Stakeholder Mapping

案例场景

一所大学正在上线一个新的 online learning platform（在线学习平台）。

利益相关者（Stakeholders）

这个项目中至少有以下几个重要的 stakeholder： 1. 学生 2. Lecturer / 教师 3. 学校管理层 4. IT 支持团队 / 开发团队 5. 外部软件供应商 6. 行政人员 7. 政府或认证机构

Power–Interest Matrix

High Power, High Interest

  • 学校管理层
  • IT project manager / 学校 IT 团队
  • 外部软件供应商

这些 stakeholder 对项目有很大的影响力，而且也非常关心项目结果，所以需要被密切管理。

High Power, Low Interest

  • 政府或认证机构
  • 高层管理者或财务负责人

这些 stakeholder 平时不一定参与项目的日常工作，但他们有审批、监督或者影响项目的权力，所以需要保持满意。

Low Power, High Interest

  • 学生
  • Lecturer / 教师
  • 行政人员

这些 stakeholder 是平台的主要使用者，所以他们对项目结果很在意，但他们本身的决策权没有那么高。对于这一类人，最重要的是及时告知，并在必要时征求意见。

Low Power, Low Interest

  • 社会公众
  • 未来可能入学的学生

这些 stakeholder 与项目的直接关系较弱，所以投入的精力可以相对少一些，但仍然需要保持关注，防止后续情况发生变化。

说明

Stakeholder mapping 的作用是帮助团队看清楚：谁的影响力最大，谁最关心项目结果，谁需要被重点管理。在这个案例里，学校管理层和 IT 团队是最核心的，而学生和教师虽然权力较低，但因为他们是实际使用者，所以也必须保持沟通。

Part C — Case Study Discussion

案例

一个 IT 团队正在为 healthcare provider 开发一款 mobile app。开发人员和测试人员之间经常发生冲突，医生抱怨自己没有被咨询，管理层也觉得项目更新不够清楚。

1. 沟通是在哪里出问题的？

这个案例里的沟通问题主要出现在几个地方。首先，开发人员和测试人员之间的沟通不顺畅，导致他们可能在软件质量、时间安排和责任划分上存在误解。其次，医生没有被充分咨询，这说明用户需求没有被及时纳入设计过程。最后，管理层没有收到清晰的项目更新，因此无法准确了解项目进展、风险以及当前需要的支持。整体来看，这个项目的问题不是单一沟通失误，而是多个沟通环节都不清楚。

2. 这里可以看到哪些 teamwork challenges？

最明显的 teamwork challenge 就是开发和测试团队之间的冲突。这通常说明合作不够、角色边界不清晰，或者成员之间缺乏信任。另一个问题是跨团队合作不足，因为医生作为重要的最终用户并没有被纳入团队讨论中。也就是说，这不仅是团队成员之间的问题，也涉及到不同专业角色之间没有形成有效配合。

3. 你会怎么做 stakeholder management？

首先要先识别关键 stakeholder，包括开发人员、测试人员、医生、管理层、healthcare provider，以及可能涉及的合规人员。然后需要根据不同 stakeholder 的需求采取不同方式。比如医生应该通过会议、访谈或反馈 session 参与进来，让 app 真正符合医疗场景。管理层则需要更规律、更清晰的进度汇报。开发人员和测试人员之间也需要建立更明确的沟通流程，比如短会、共享任务板和统一的问题反馈机制。

4. Communication、teamwork 和 stakeholder management 如何结合起来解决问题？

这三者其实是连在一起的。良好的 communication 可以让信息传递更及时、更清楚，减少误解。有效的 teamwork 可以增强成员之间的合作、信任和责任感。合理的 stakeholder management 则能确保像医生和管理层这样的重要角色真正参与决策过程。如果三者一起改善，就可以减少冲突、提高协作效率，也更有可能交付一个真正满足需求的 healthcare app。

总结

这一周的 tutorial 说明了 communication、teamwork 和 stakeholder management 在 IT 项目里是紧密相关的。如果沟通不好，团队就容易混乱；如果团队合作差，冲突就会增加；如果忽略 stakeholder，做出来的系统就可能不符合实际需求。只有把这三个方面一起做好，项目才更容易成功。
